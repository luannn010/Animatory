System Architecture · v0.1 draft

Ptolemy Orchestrating many workers through isolated sandboxes — cheap parallel verification, then a single serial check against reality.

A small model assigns coding tasks to 4–8 workers running at once, each in its own git-worktree sandbox. Workers develop against HTTP mocks in parallel, then survivors test serially against the real environment before merging. Every merge propagates back to the workers still in flight.

Principle

Keep the model out of the plumbing. Git, Docker, locks and tests are deterministic Go. The model issues intents on top — so any failure is attributable to a decision or a controller bug, never a tangle of both.

02

Plan phase · two agents

A task agent defines and splits the work; a resource agent hands each worker its mock plus failing unit tests. Because a different agent writes the tests, the worker cannot game tests it authored itself — the tests become an external contract.

Task agent
define + split sub-tasks
Resource agent
mock API + failing tests
Work package · per worker
sub-task · mock · red tests
Why split

Separating test-author from implementer is the main defense against an LLM coding to satisfy the mock rather than the real contract. The resource agent's failing tests are the spec the worker must earn.

03

Execution · the TDD loop

Inside each sandbox the worker runs red → green → refactor against the mock. On a failed integration test, the gap is captured as a new failing test before looping back — this gives the loop a memory so it can't spin forever.

Loop memory

A failed Stage 2 means the mock drifted or a real bug slipped through. Encode it as a new red test, then go green again — each pass through the loop is strictly more constrained than the last. That same regression test also signals the mock-update module to fix the mock for other workers.

Entry ticket

Before a worker may acquire the integration lock it must be current with base. If base moved while it waited in the queue, it is bounced back to sync + re-test first — so the one real environment never runs on stale code.

05

Relevance filter · sender & receiver

The sender broadcasts; each worker's receiver decides whether the change touches its work. This turns a global stall into a targeted one — only workers whose scope intersects the change ever pause.

Yes → update mock

Asymmetry

A false positive wastes one re-test; a false negative lets a worker code against a stale mock and fail expensively at Stage 2. So err toward resyncing — tune the filter slightly too eager, never too lazy. Relevance is computed deterministically from the worker's loaded-mock set, not a model judgment.

07

Open decisions & status

The architecture is sound; these are the policy choices it implies but hasn't fixed. Closing them is the prerequisite to building.

High impact

  • Code conflict policy — human resolves, or worker re-tasked?
  • Schema / migration conflicts — allow in v1, or defer?
  • Mock generation mode — derive-from-contract for v1; re-capture later

Prevent silent breakage

  • Integration entry-ticket — current with base before the lock
  • Lock lease / timeout — crashed worker can't hold the env
  • Real-env state reset — who resets the DB between serial runs

Deferred (name now)

  • Review UI — live app + diff + logs + FEATURE.md
  • Worker crash / retry semantics
  • Secrets in the real environment — a security boundary
Build order

Worktree + stack lifecycle → review artifacts → mock layer + Stage 1 → integration lock + Stage 2 → base_updated propagation → relevance receiver → wire in the model last, on proven plumbing.
